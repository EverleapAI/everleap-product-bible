# Onboarding

## Purpose

Onboarding is the beginning of the Everleap relationship.

Its purpose is not to collect data.

Its purpose is to establish trust.

By the time onboarding is complete, the user should feel that Everleap understands how to ask thoughtful questions—not that it already understands them.

The goal is curiosity.

Not analysis.

---

# The Problem

Most onboarding experiences are administrative.

They collect:

- name
- email
- password
- demographics

Everleap is different.

The first experience should immediately begin helping the user think about themselves.

Every question should feel like it exists for the user's benefit rather than the application's benefit.

---

# Primary Goals

Onboarding has five goals.

## Goal 1 — Build Trust

The user should believe:

"This feels different."

Everleap should immediately demonstrate humility.

It should avoid making claims.

It should avoid pretending to know the user.

Instead it should create confidence that this will become an interesting journey.

---

## Goal 2 — Establish Conversation

Onboarding should feel like a conversation.

Not an interview.

Questions should flow naturally.

Transitions should feel intentional.

The user should never feel like they are filling out a survey.

---

## Goal 3 — Collect Initial Evidence

Every answer contributes evidence.

At this stage, evidence is intentionally limited.

Everleap should avoid making meaningful conclusions.

Instead it begins creating the foundation for future understanding.

---

## Goal 4 — Create Momentum

The end of onboarding should create excitement.

The user should want to continue.

The experience should naturally lead into Story.

The user should never feel:

"I'm done."

Instead they should feel:

"I'm just getting started."

---

## Goal 5 — Personalize Tomorrow

Onboarding exists primarily so Tomorrow can be better than Today.

The information collected becomes the initial context for:

- Today
- Story
- Insights
- Recommendations

The value compounds over time.

---

# What Onboarding Should Never Do

Onboarding should never:

- assign personality types
- recommend careers
- generate deep insights
- summarize the user
- predict the future

There is not enough evidence.

Pretending otherwise damages trust.

---

# User Experience

The user should experience onboarding as:

Curious.

Friendly.

Thoughtful.

Comfortable.

Respectful.

Hopeful.

They should never feel evaluated.

---

# AI During Onboarding

Artificial intelligence should ask good questions.

Not provide good answers.

AI exists to:

- keep the conversation natural
- encourage reflection
- clarify when needed
- maintain momentum

It should resist the temptation to interpret.

Interpretation belongs later.

---

# Relationship to Story

Onboarding is intentionally shallow.

Story is intentionally deeper.

Onboarding discovers where to begin.

Story discovers why.

The two experiences should feel connected.

Not duplicated.

---

# Success

Onboarding succeeds when a user finishes thinking:

"I want to know what comes next."

Not:

"I wonder what score I got."

Curiosity is the desired outcome.

Not completion.

---

# Product Rules

Onboarding should always:

- Build trust before collecting information.
- Ask before explaining.
- Encourage before evaluating.
- Leave the user wanting to continue.
- Create enough evidence to personalize the next experience, but never enough to claim understanding.

Onboarding is the first conversation.

It should feel like the beginning of a relationship, not the completion of an assessment.
