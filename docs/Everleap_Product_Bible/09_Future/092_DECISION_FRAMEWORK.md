# Product Decision Framework

## Purpose

Every successful product eventually reaches a point where there are more good ideas than time to build them.

The purpose of this document is to ensure Everleap makes those decisions consistently.

This framework exists to answer one question:

> **"Should we build this?"**

Not:

> "Can we build this?"

Those are very different questions.

---

# The Guiding Principle

Every feature should help users:

- Understand themselves better.
- Reflect more deeply.
- Take meaningful action.

If a feature does not improve one of those three outcomes, it should probably not exist.

---

# The Five Questions

Before any feature begins development, it should answer five questions.

## 1. What problem does this solve?

Every feature should solve a real user problem.

Not an engineering problem.

Not an AI problem.

Not a "this would be cool" problem.

If the problem cannot be clearly described in one sentence, the feature is probably not ready.

---

## 2. Why now?

Timing matters.

Some ideas are excellent but premature.

Ask:

- Does this unblock other work?
- Will users immediately benefit?
- Is there enough evidence to build it well?

If the answer is no, it belongs on the roadmap rather than in development.

---

## 3. Does it strengthen trust?

Trust is Everleap's most valuable asset.

Every feature should increase trust.

Never reduce it.

Ask:

- Is it explainable?
- Is it evidence-based?
- Is it intellectually honest?
- Could it accidentally overstate certainty?

If trust decreases, the feature fails.

---

## 4. Does it simplify or complicate?

Every feature introduces complexity.

Ask:

- Does this reduce cognitive load?
- Does it make the product easier to understand?
- Does it remove friction?

If complexity grows faster than value, don't build it.

---

## 5. Can it improve over time?

Everleap should become smarter through use.

Features that learn are more valuable than features that remain static.

Ask:

- Does this create new evidence?
- Does it improve future recommendations?
- Does it help us better understand the user?

If a feature never becomes more valuable after launch, reconsider it.

---

# The Four Filters

Every feature should pass four filters.

## User Filter

Does this genuinely help the user?

Not theoretically.

Actually.

---

## Product Filter

Does it align with Everleap's philosophy?

Or does it pull the product toward becoming something else?

---

## Architecture Filter

Does it fit the architecture?

Or does it require breaking core principles?

Good features rarely require architectural compromises.

---

## Science Filter

Can the feature be grounded in evidence?

Or would it require AI to invent observations?

Evidence should always come first.

---

# Things We Don't Build

There are entire categories of features that Everleap intentionally avoids.

Examples include:

- engagement for engagement's sake
- infinite feeds
- streaks that create guilt
- vanity metrics
- social comparison
- leaderboards
- addictive mechanics
- clickbait notifications

Growth is the goal.

Attention is not.

---

# The "One Year Later" Test

Imagine this feature has existed for one year.

Would users say:

"I'm glad this exists."

Or:

"I barely notice it."

Great features quietly become essential.

Flashy features usually become forgotten.

---

# The "Remove It" Test

Ask:

"If we removed this feature tomorrow, would users genuinely miss it?"

If the answer is no...

it probably wasn't solving an important problem.

---

# Technical Elegance

Engineers should ask:

Can this be explained simply?

If implementation requires excessive exceptions, special cases, or hidden behavior...

the design probably needs more work.

Simple systems survive.

Complicated systems decay.

---

# Product Debt

Not all debt is technical.

Products also accumulate:

- UX debt
- prompt debt
- design debt
- content debt
- science debt

Before adding something new, ask whether improving an existing feature creates more value.

Often it does.

---

# Success

The best Everleap features share common characteristics.

They are:

- calm
- understandable
- evidence-based
- explainable
- durable
- deeply useful

They rarely feel revolutionary.

They simply feel right.

---

# Product Rules

Before building anything, ask:

1. Does this improve understanding?
2. Does this encourage reflection?
3. Does this support action?
4. Does this strengthen trust?
5. Will this still feel valuable five years from now?

If the answer to most of those questions is yes, build it.

If not, keep thinking.

Everleap should never become the product with the most features.

It should become the product with the most thoughtful ones.
